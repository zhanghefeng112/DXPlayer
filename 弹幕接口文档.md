## 一、基础信息

| 项目 | 说明 |
| --- | --- |
| 基础地址 | `http://{host}:8666`（默认端口 8666） |
| 协议 | HTTP/1.1、WebSocket |
| Content-Type | `application/json`（推荐）或 `application/x-www-form-urlencoded` |
| 字符编码 | UTF-8（支持 emoji） |

---

## 二、认证机制

### 2.1 Token 传递方式

| 场景 | 传递方式 | 示例 |
| --- | --- | --- |
| HTTP 接口 | Header `X-Auth-Token` | `X-Auth-Token: eyJhbGciOi...` |
| WebSocket | Query 参数 `token` | `ws://host:8666/ws/room/1001?token=eyJhbGciOi...` |

### 2.2 JWT 规范

- **签名算法：** HMAC-SHA256
- **Payload 必须包含：**

```json
{
  "oauth_id": "wx_12345",
  "oauth_type": "wechat"
}
```

### 2.3 哪些接口需要认证

| 接口 | 是否需要认证 |
| --- | --- |
| `POST /api/danmu/send` | **需要** |
| `GET /api/danmu/video` | 不需要 |
| `GET /api/danmu/live` | 不需要 |
| `WS /ws/room/:room_id` | **需要** |
| `WS /ws/video/:video_id` | **需要** |

### 2.4 Debug 模式（开发联调）

当服务端运行在 `debug` 模式时：

- 不校验 JWT 签名
- 可通过请求参数直接传递 `oauth_id` 和 `oauth_type`
  - HTTP：JSON Body 或 Form 字段
  - WebSocket：Query 参数 `oauth_id`，如 `ws://host:8666/ws/room/1001?oauth_id=test_user`
- 未传 `oauth_id` 仍会返回 `errno: 10008`

### 2.5 防篡改机制

弹幕发送接口（`POST /api/danmu/send`）中，服务端从 JWT Token 解析 `oauth_id`，与请求体中的 `oauth_id` 做强一致性校验。**两者不一致将被拒绝**，返回 `errno: 30105`。

## 三、统一响应格式

所有 HTTP 接口返回统一 JSON 结构：

```json
{
  "errno": 0,
  "errmsg": "OK",
  "data": null
}
```

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| errno | int | 错误码，`0` 表示成功 |
| errmsg | string | 错误描述 |
| data | any | 业务数据，无数据时为 `null`；列表接口返回 `{"list": [], "total": 0}` 结构 |

**列表接口统一返回结构：**

```json
{
  "errno": 0,
  "errmsg": "OK",
  "data": {
    "list": [],
    "total": 0
  }
}
```

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| data.list | array | 数据列表，始终为数组，无数据时为 `[]` |
| data.total | int64 | 符合条件的总条数 |

## 四、错误码表

### 通用错误码（10xxx）

| errno | 说明 | 常见原因 |
| --- | --- | --- |
| 10001 | 内部错误 | 服务端异常 |
| 10002 | 参数无效 | 请求参数格式错误、绑定失败 |
| 10005 | 参数错误 | 必填参数缺失或值不合法 |
| 10007 | 认证失败 | JWT 签名无效或过期 |
| 10008 | 未认证 | 未传 Token 或 Token 中缺少 oauth_id |
| 10010 | 未登录 | 需登录才能操作 |
| 10011 | 权限不足 | 无操作权限 |

### 通用服务端异常（101xx）

| errno | 说明 | 处理建议 |
| --- | --- | --- |
| 10101 | 弹幕发送失败 | 提示用户稍后重试 |
| 10102 | 弹幕查询失败 | 提示用户稍后重试 |
| 10103 | WebSocket 升级失败 | 检查网络或代理配置 |
| 10104 | 屏蔽用户失败 | 提示管理员稍后重试 |
| 10105 | 解除屏蔽失败 | 提示管理员稍后重试 |
| 10106 | 用户设置更新失败 | 提示用户稍后重试 |
| 10107 | 用户信息同步失败 | 检查请求参数或稍后重试 |

### 数据错误码（20xxx）

| errno | 说明 |
| --- | --- |
| 20002 | 数据库错误 |
| 20004 | 数据不存在 |

### 弹幕业务错误码（301xx）

| errno | 说明 | 处理建议 |
| --- | --- | --- |
| 30102 | 发送过于频繁 | 前端做节流，提示用户"发送太快，请稍后再试" |
| 30103 | 重复弹幕 | 提示用户"请勿发送重复内容" |
| 30104 | 弹幕包含敏感词 | 提示用户修改内容 |
| 30105 | 用户身份校验失败 | Token 中的 oauth_id 与请求体不一致，检查前端传参 |
| 30107 | 弹幕类型无效 | `type` 只允许 `live` 或 `video` |
| 30109 | 弹幕模式无效 | `mode` 只允许 `scroll`/`top`/`bottom`/`reverse` |
| 30110 | 弹幕颜色格式无效 | `color` 必须为 `#RRGGBB` 格式 |
| 30111 | 该颜色为保留色 | 管理员红色 `#FF0000` / UP主黄色 `#FFD700` 为系统预制保留色 |
| 30112 | 用户已被屏蔽 | 联系管理员解除屏蔽 |
| 30116 | 非VIP不可使用该颜色 | 该颜色属于VIP等级色，需VIP等级才能使用 |
| 30117 | 过滤模式无效 | `filter` 只允许 `none`/`uper`/`friend` |

## 五、HTTP 接口

### 5.1 弹幕发送

```
POST /api/danmu/send
```

**认证：** 需要（Header `X-Auth-Token`）

**请求参数：**

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| type | string | 是 | 场景类型：`live`（直播）/ `video`（点播） |
| oauth_id | string | 是 | 用户 OAuth ID（须与 Token 中的一致） |
| oauth_type | string | 否 | OAuth 类型（如 `wechat`、`github`） |
| content | string | 是 | 弹幕内容，最大 255 字符 |
| room_id | int64 | type=live 时必填 | 直播房间 ID，必须大于 0 |
| video_id | int64 | type=video 时必填 | 视频 ID，必须大于 0 |
| video_time | int64 | type=video 时必填 | 当前播放位置（毫秒），不能为负 |
| mode | string | 否 | 弹幕模式：`scroll`(滚动)/`top`(顶部固定)/`bottom`(底部固定)/`reverse`(从左往右)，默认 `scroll` |
| color | string | 否 | 弹幕颜色，HEX 格式如 `#FFFFFF`，默认空（服务端按用户等级解析） |

**Content-Type** 支持 `application/json` 和 `application/x-www-form-urlencoded`。

#### 直播弹幕请求示例

```bash
curl -X POST http://127.0.0.1:8666/api/danmu/send \
  -H "Content-Type: application/json" \
  -H "X-Auth-Token: eyJhbGciOi..." \
  -d '{
    "type": "live",
    "oauth_id": "wx_12345",
    "oauth_type": "wechat",
    "room_id": 1001,
    "content": "666",
    "mode": "scroll",
    "color": "#4FC3F7"
  }'
```

#### 点播弹幕请求示例

```bash
curl -X POST http://127.0.0.1:8666/api/danmu/send \
  -H "Content-Type: application/json" \
  -H "X-Auth-Token: eyJhbGciOi..." \
  -d '{
    "type": "video",
    "oauth_id": "wx_12345",
    "oauth_type": "wechat",
    "video_id": 2001,
    "video_time": 102500,
    "content": "前方高能",
    "mode": "top",
    "color": "#FFFFFF"
  }'
```

#### 成功响应

```json
{
  "errno": 0,
  "errmsg": "OK",
  "data": null
}
```

#### 错误响应示例

```json
{
  "errno": 30102,
  "errmsg": "发送过于频繁",
  "data": null
}
```

#### 发送链路说明

弹幕发送经过 8 步校验链：

1. **UID 防篡改** — Token oauth_id 与请求体 oauth_id 一致性校验
2. **屏蔽检查** — 检查用户是否被屏蔽（Redis 缓存 → DB 降级）
3. **参数校验** — 必填字段、值范围检查
4. **模式/颜色校验** — mode 枚举验证、color HEX 格式 + 保留色 + VIP 颜色权限
5. **频率限流** — 每秒最多发送 5 条（服务端配置 `danmu.rate_limit`）
6. **内容去重** — 同一用户相同内容在 5 秒内不可重复发送（`danmu.dedup_ttl`）
7. **敏感词过滤** — Trie 树匹配
8. **分发写入** — 写入 Redis 后立即返回，数据库异步落库

### 5.2 点播弹幕查询

```
GET /api/danmu/video
```

**认证：** 不需要

**请求参数（Query）：**

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| video_id | int64 | 是 | 视频 ID |
| time | int64 | 是 | 当前播放位置（毫秒） |
| sec_limit | int | 否 | 每秒最多返回弹幕条数，默认不限制 |
| filter | string | 否 | 过滤模式：`none`(全部)/`uper`(只看UP主)/`friend`(只看好友)，默认 `none` |
| simplify | int | 否 | 精简模式：`0`(全量)/`1`(精简合并重复弹幕)，默认 `0` |

**查询范围：** 返回 `[time, time + 10000]`（即当前时间点往后 10 秒）内的弹幕。

#### 请求示例

```bash
curl "http://127.0.0.1:8666/api/danmu/video?video_id=2001&time=102500&sec_limit=50"
```

#### 成功响应

```json
{
  "errno": 0,
  "errmsg": "OK",
  "data": {
    "list": [
      {
        "dmid": "WjxeHZ9LLoptfQ",
        "oauth_id": "wx_12345",
        "content": "哈哈哈",
        "mode": "scroll",
        "color": "#FFD700",
        "title": "UP主",
        "video_time": 102500,
        "send_time": 1713000001234
      },
      {
        "dmid": "X1yeHZ9LLoptgA",
        "oauth_id": "wx_67890",
        "content": "前方高能",
        "mode": "top",
        "color": "#4FC3F7",
        "title": "VIP5",
        "video_time": 103200,
        "send_time": 1713000002500
      }
    ],
    "total": 128
  }
}
```

**注意：** `data.list` 始终为数组，无数据时返回 `[]`；`data.total` 为符合条件的总条数。

#### sec_limit 说明

`sec_limit` 用于弹幕密度控制。当某一秒内弹幕过多时，只返回前 `sec_limit` 条，避免弹幕过密影响播放体验。按 `video_time` 的秒级分桶计数。

## 六、WebSocket 接口

### 6.1 直播 WebSocket

```
WS /ws/room/:room_id?token=xxx
```

**认证：** 需要（Query 参数 `token`；debug 模式可用 `oauth_id`）

**通信模式：** 服务端主动推送，客户端只需保持连接。

#### 连接流程

```
1. 客户端发起 WS 连接请求，携带 token
2. 服务端验签（失败返回 HTTP 错误，不升级为 WS）
3. 验签成功 → 升级为 WS 连接
4. 服务端持续推送该房间的实时弹幕
```

#### 连接示例

```javascript
// Token 认证
const ws = new WebSocket('ws://127.0.0.1:8666/ws/room/1001?token=eyJhbGciOi...');

// Debug 模式
const ws = new WebSocket('ws://127.0.0.1:8666/ws/room/1001?oauth_id=test_user');
```

#### 服务端推送格式

```json
{
  "type": "danmu",
  "data": {
    "list": [
      {
        "dmid": "WjxeHZ9LLoptfQ",
        "oauth_id": "wx_12345",
        "content": "666",
        "mode": "scroll",
        "color": "#FFD700",
        "title": "UP主",
        "send_time": 1713000001234
      }
    ],
    "total": 1
  }
}
```

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| type | string | 固定为 `"danmu"` |
| data.list | array | 弹幕数组，**始终为数组**（支持批量合并推送） |
| data.total | int | 本次推送弹幕条数 |

#### 心跳机制

- 服务端每 **54 秒** 发送一次 Ping 帧（`pong_wait` 的 90%）
- 客户端需在 **60 秒** 内回复 Pong（浏览器 WebSocket API 自动处理）
- 超时未收到 Pong，服务端将关闭连接

#### 注意事项

- 客户端无需发送任何消息，保持连接即可接收推送
- 连接断开后前端应实现重连逻辑
- 同一房间多个客户端各自收到一份推送，不会重复

### 6.2 点播 WebSocket

```
WS /ws/video/:video_id?token=xxx
```

**认证：** 需要（Query 参数 `token`；debug 模式可用 `oauth_id`）

**通信模式：** 双向通信 — 客户端上报播放进度，服务端按进度推送弹幕。

#### 连接流程

```
1. 客户端发起 WS 连接请求，携带 token
2. 服务端验签（失败返回 HTTP 错误，不升级为 WS）
3. 验签成功 → 升级为 WS 连接
4. 客户端持续上报播放进度 video_time
5. 服务端查询该时间窗口弹幕并推送
6. 其他用户发送的实时弹幕，按客户端进度匹配后推送
```

#### 连接示例

```javascript
const ws = new WebSocket('ws://127.0.0.1:8666/ws/video/2001?token=eyJhbGciOi...');
```

#### 客户端上报格式（Client → Server）

客户端需**持续上报**当前播放进度，建议每秒上报一次：

```json
{"video_time": 102500}
```

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| video_time | int64 | 是 | 当前播放位置（毫秒） |

**前端参考实现：**

```javascript
// 每秒上报播放进度
setInterval(() => {
  if (ws.readyState === WebSocket.OPEN) {
    const currentTime = Math.floor(videoElement.currentTime * 1000);
    ws.send(JSON.stringify({ video_time: currentTime }));
  }
}, 1000);
```

#### 服务端推送格式（Server → Client）

```json
{
  "type": "danmu",
  "data": {
    "list": [
      {
        "dmid": "WjxeHZ9LLoptfQ",
        "oauth_id": "wx_12345",
        "content": "哈哈哈",
        "mode": "scroll",
        "color": "#FFFFFF",
        "title": "VIP3",
        "video_time": 102500,
        "send_time": 1713000001234
      }
    ],
    "total": 1
  }
}
```

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| type | string | 固定为 `"danmu"` |
| data.list | array | 弹幕数组，**始终为数组** |
| data.total | int | 本次推送弹幕条数 |

#### 弹幕推送规则

**历史弹幕：** 每次客户端上报 `video_time` 后，服务端查询 `[video_time, video_time + 1000]`（1 秒窗口）内的弹幕并推送。

**实时弹幕：** 其他用户在同一视频上发送弹幕时，服务端比较弹幕的 `video_time` 与当前客户端上报的播放进度，差值在 **3 秒**（`danmu.video_time_window: 3000ms`）以内才下发。

#### 心跳机制

与直播 WS 相同，服务端 54 秒发 Ping，客户端需 60 秒内回 Pong。

#### 注意事项

- **必须上报进度才能收到弹幕**，不上报则不推送
- 拖动进度条后应立即上报新的 `video_time`
- 暂停播放时可停止上报，恢复播放后继续上报

## 七、通用字段说明

### 7.1 弹幕对象（DanmuItem）

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| dmid | string | 弹幕唯一标识（12 字符），可用于客户端消息去重 |
| oauth_id | string | 发送者 OAuth ID |
| content | string | 弹幕内容 |
| mode | string | 弹幕模式：`scroll`/`top`/`bottom`/`reverse` |
| color | string | 弹幕颜色：HEX 格式如 `#FFFFFF`（管理员强制 `#FF0000`，UP主强制 `#FFD700`） |
| title | string | 发送者头衔：`GM`=管理员 / `UP主`=UP主 / `VIP1`~`VIP10`=VIP用户 / `""`=普通用户 |
| video_time | int64 | 视频时间轴位置（毫秒），**仅点播弹幕有此字段** |

| count | int | 合并数量，**仅精简模式（simplify=1）时出现**，值>1表示该弹幕是多条相同内容合并后的结果 |

### 7.2 dmid 说明

`dmid` 是 12 字符的 base64url 编码字符串，作为弹幕的唯一标识。

**用途：**
- 客户端消息去重（WS 推送 + HTTP 查询可能有重叠）
- 问题排查（可反推来源）

**前端使用建议：** 在渲染弹幕前，维护一个已渲染 `dmid` 的 Set，跳过重复项。

```javascript
const renderedDmids = new Set();

function onDanmuReceived(msg) {
  msg.data.list.forEach(item => {
    if (renderedDmids.has(item.dmid)) return;
    renderedDmids.add(item.dmid);
    renderDanmu(item);
  });
}
```

| --- | --- | --- |
| video_time | 毫秒 | 视频播放位置，从 0 开始 |
| send_time | 毫秒 | Unix 时间戳（如 `1713000001234`） |

## 八、前端集成指南

### 8.1 直播场景接入流程

```
1. 用户进入直播间
2. [可选] GET /api/danmu/live 拉取历史弹幕
3. 建立 WS 连接 /ws/room/{room_id}?token=xxx
4. 监听 WS 推送，渲染弹幕
5. 用户发弹幕 → POST /api/danmu/send (type=live)
6. 离开时关闭 WS 连接
```

### 8.2 点播场景接入流程

```
1. 用户打开视频页
2. [可选] GET /api/danmu/video 预加载弹幕（如前 10 秒）
3. 建立 WS 连接 /ws/video/{video_id}?token=xxx
4. 播放器开始播放后，每秒上报 video_time
5. 监听 WS 推送，按 video_time 渲染弹幕
6. 用户发弹幕 → POST /api/danmu/send (type=video)
7. 拖动进度条 → 立即上报新 video_time
8. 离开时关闭 WS 连接
```

### 8.3 WS 重连策略

建议实现指数退避重连：

```javascript
let reconnectDelay = 1000; // 初始 1 秒
const maxDelay = 30000;    // 最大 30 秒

function connect() {
  const ws = new WebSocket(url);

  ws.onopen = () => {
    reconnectDelay = 1000; // 连接成功后重置
  };

  ws.onclose = () => {
    setTimeout(() => {
      reconnectDelay = Math.min(reconnectDelay * 2, maxDelay);
      connect();
    }, reconnectDelay);
  };
}
```

### 8.4 频率限制说明

| 限制项 | 阈值 | 说明 |
| --- | --- | --- |
| 发送频率 | 每秒 5 条 | 超出返回 `errno: 30102` |
| 内容去重 | 5 秒内不可重复 | 超出返回 `errno: 30103` |
| 弹幕长度 | 最大 255 字符 | 超出返回 `errno: 10002` |
| WS 消息大小 | 最大 512 字节 | 超出服务端断开连接 |

**前端建议：** 发送按钮做 200ms 节流，输入框做最大长度限制。

## 十一、三方用户信息同步接口（v0.2.0 新增）

### 11.1 同步用户信息

```
POST /api/danmu/sync/user
```

**认证：** 需要

**说明：** 用于从三方系统同步用户信息到弹幕服务，支持 UPSERT（存在则更新，不存在则插入）。`oauth_type`、`vip_level`、`is_uper` 三个字段均可通过此接口同步。后续对接三方系统时，由三方系统调用此接口推送用户信息变更。

**请求参数：**

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| oauth_id | string | 是 | OAuth 用户 ID |
| oauth_type | string | 否 | OAuth 类型（如 `wechat`、`github`） |
| vip_level | int | 否 | VIP 等级：`0`=非VIP，`1-10`=VIP 等级 |
| is_uper | int | 否 | 是否UP主：`0`=否，`1`=是 |

#### 请求示例

```bash
curl -X POST http://127.0.0.1:8666/api/danmu/sync/user \
  -H "Content-Type: application/json" \
  -H "X-Auth-Token: eyJhbGciOi..." \
  -d '{
    "oauth_id": "wx_12345",
    "oauth_type": "wechat",
    "vip_level": 5,
    "is_uper": 1
  }'
```

#### 成功响应

```json
{"errno": 0, "errmsg": "OK", "data": null}
```

#### 业务逻辑

1. 校验 `oauth_id` 必填、`vip_level` 范围 0-10、`is_uper` 值为 0 或 1
2. UPSERT `dm_user_setting` 表：存在则更新 `oauth_type`/`vip_level`/`is_uper`，不存在则插入
3. 更新后清除 Redis 用户设置缓存（`danmu:setting:{oauth_id}`），下次读取时重建

#### 扩展说明

当前接口已具备完整的同步能力，三方系统对接时：
- 用户注册/信息变更时调用此接口推送
- 批量同步场景可循环调用（后续可扩展批量接口）
- `is_uper` 变更后立即影响"只看UP主"过滤和 UP 主黄色权限
